💡 The Core Idea

Imagine you're furnishing an apartment. You could buy things one by one, keep track in your head, and forget what you ordered. Or you write a shopping list — and whenever you update the list, everything adjusts automatically. Terraform is that shopping list, but for cloud infrastructure (servers, databases, networks…). You describe what you want, and Terraform figures out how to make it happen.

📄 Configuration Files (.tf)

You write plain text files in HCL (HashiCorp Configuration Language) — it's designed to be human-readable. You describe your infrastructure declaratively: "I want an EC2 server of this size in this region." You don't write steps — you just describe the end state.

# main.tf — "I want an EC2 server" resource "aws_instance" "web_server" { ami = "ami-0c55b159cbfafe1f0" instance_type = "t2.micro" tags = { Name = "MyWebServer" } }

🗺️ The State File

Terraform stores a terraform.tfstate file that records what it has already built. This is crucial — it's how Terraform knows the difference between "create this new server" vs "this server already exists, leave it alone."

Think of it like a ledger. Every time you run Terraform, it compares your config (what you want) against the state file (what exists) and only makes the changes needed. In teams, this state file is usually stored in a shared location like S3 so everyone's in sync.

🔌 Providers

Terraform itself doesn't know how to create an AWS server or a Google Cloud database. Providers are plugins that add that knowledge. There are thousands — AWS, Azure, GCP, Kubernetes, GitHub, Cloudflare, even Spotify.

terraform { required_providers { aws = { source = "hashicorp/aws" version = "~> 5.0" } } } provider "aws" { region = "us-east-1" }

📦 Resources

A resource is the most basic building block in Terraform — it represents one piece of infrastructure. An S3 bucket, a virtual machine, a DNS record, a firewall rule. You combine many resources to describe a full system.

Resources have a type (e.g. aws_s3_bucket), a name you choose, and arguments that configure it. Resources can also reference each other.

# A bucket that references a variable resource "aws_s3_bucket" "my_bucket" { bucket = var.bucket_name } # A policy that references the bucket above resource "aws_s3_bucket_policy" "policy" { bucket = aws_s3_bucket.my_bucket.id }

♻️ Modules

When you find yourself writing the same infrastructure patterns repeatedly, you wrap them in a module — a reusable package. Instead of defining 10 resources every time you need a web server setup, you call a module with a few inputs.

The Terraform Registry has thousands of community-built modules (e.g. a complete VPC, an EKS cluster) you can use out of the box — like npm for infrastructure.

# Using a community VPC module module "vpc" { source = "terraform-aws-modules/vpc/aws" version = "5.0.0" name = "my-vpc" cidr = "10.0.0.0/16" }
// The Terraform Workflow
01

Write your config

Describe what infrastructure you want in .tf files. No imperative steps — just the desired end state.

📝 Edit main.tf
02

terraform init

Downloads the required provider plugins and sets up the working directory. Run this once per project (or after adding providers).

$ terraform init
03

terraform plan

Terraform shows you a preview — "here's what I'll create, change, or destroy" — without actually doing anything. Always review this first!

$ terraform plan
04

terraform apply

Executes the plan. Terraform calls the cloud APIs to create/update/delete resources. When done, your infrastructure matches your config.

$ terraform apply
// Terraform vs. Doing It Manually
Aspect Manual (Console / CLI) With Terraform
Reproducibility Hard — "What did I click again?" Easy — re-run the same config
Team collaboration Risky — who changed what? Safe — config lives in Git
Multiple environments Tedious — redo for dev/staging/prod Simple — reuse with variables
Disaster recovery Slow — recreate from memory Fast — terraform apply restores
Change visibility None — no diff, no audit trail Full — plan shows exact changes
